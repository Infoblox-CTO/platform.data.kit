# Quickstart: Canonical Lock & Catalog

This guide walks through the lock-and-catalog workflow for a DataKit package.

## Prerequisites

- `dk` CLI installed
- OCI registry accessible (e.g., local k3d registry via `dk dev up`)
- An existing DataKit package with a `dk.yaml`

## 1. Lock Dependencies

Given a `dk.yaml` with version constraints:

```yaml
apiVersion: datakit.infoblox.dev/v1alpha1
kind: Store
metadata:
  name: my-store
  namespace: demo
spec:
  connector: postgres-cdc
  connectorVersion: ">=1.0.0"
  connection:
    host: "${PG_HOST}"
```

Run `dk lock` to resolve and pin all dependencies:

```bash
$ dk lock

Resolving dependencies...
  postgres-cdc: >=1.0.0 → 1.2.3 (sha256:abc123...)
Lock file written: dk.lock (1 dependency)
```

This creates `dk.lock`:

```yaml
# Auto-generated by dk lock. DO NOT EDIT.
lockVersion: 1
manifestHash: "sha256:e3b0c44298fc..."
generatedAt: "2026-03-07T12:00:00Z"
dependencies:
  - name: postgres-cdc
    kind: Connector
    version: "1.2.3"
    registry: ghcr.io/infoblox-cto
    digest: "sha256:abc123def456..."
    constraint: ">=1.0.0"
```

## 2. Build With Locked Versions

```bash
$ dk build

Using locked dependencies from dk.lock
  postgres-cdc: 1.2.3 (sha256:abc123...)
Build complete.
```

The build uses the pinned version — even if `1.3.0` has been published since.

## 3. Validate Lock Consistency

```bash
$ dk lint

✓ dk.yaml: valid
✓ dk.lock: consistent with dk.yaml
✓ schemas/: valid
```

If you edit `dk.yaml` without re-locking:

```bash
$ dk lint

✓ dk.yaml: valid
✗ dk.lock: [E300] manifest hash mismatch — run 'dk lock' to update
```

## 4. Update Locked Dependencies

Update all dependencies:

```bash
$ dk lock --update

Resolving dependencies...
  postgres-cdc: >=1.0.0 → 1.3.0 (sha256:def789...)
Lock file updated: dk.lock (1 dependency changed)
```

Update a single dependency:

```bash
$ dk lock --update postgres-cdc

Resolving dependencies...
  postgres-cdc: >=1.0.0 → 1.3.0 (sha256:def789...)
Lock file updated: dk.lock (1 dependency changed)
```

## 5. Publish With Catalog Metadata

```bash
$ dk publish

Publishing my-store:0.1.0 to ghcr.io/infoblox-cto...
  ✓ Artifact pushed (sha256:aaa111...)
  ✓ Catalog entry created
    Kind: Store
    Owner: demo-team
    Dependencies: postgres-cdc@1.2.3
Done.
```

The publish command automatically populates catalog metadata from your manifest.

## 6. Search the Catalog

```bash
$ dk catalog search --kind Connector

NAME           VERSION  KIND        NAMESPACE  OWNER          TAGS
postgres-cdc   1.2.3    Connector   infoblox   platform-team  production,certified
postgres-cdc   1.3.0    Connector   infoblox   platform-team  production
mysql-cdc      2.0.0    Connector   infoblox   platform-team  production
```

Filter by tag:

```bash
$ dk catalog search --tag certified

NAME           VERSION  KIND        NAMESPACE  OWNER          TAGS
postgres-cdc   1.2.3    Connector   infoblox   platform-team  production,certified
```

## Workflow Summary

```
dk init          → Create dk.yaml
dk lock          → Resolve constraints → Write dk.lock
dk build         → Build using pinned versions
dk lint          → Validate lock ↔ manifest consistency
dk lock --update → Re-resolve to latest satisfying versions
dk publish       → Push artifact + catalog entry to registry
dk catalog search→ Discover available packages
```

## Key Concepts

| Concept | File/Location | Description |
|---------|--------------|-------------|
| Version constraint | `dk.yaml` | Declares acceptable version range (e.g., `>=1.0.0`) |
| Locked dependency | `dk.lock` | Pins exact version + digest for reproducibility |
| Catalog entry | OCI registry | Machine-readable metadata for discovery |
| Manifest hash | `dk.lock` | Detects when dk.yaml changes require re-locking |
