# Data Model: Python CloudQuery Plugin Support

**Feature**: 010-python-cloudquery-plugins
**Date**: 2026-02-13

## Entities

### Python Plugin Project (directory structure)

A Python CloudQuery plugin project is a directory containing:

```
<name>/
├── dp.yaml                          # Package manifest
├── main.py                          # Entry point (gRPC serve)
├── requirements.txt                 # pip dependencies
├── pyproject.toml                   # Python project metadata
├── plugin/
│   ├── __init__.py
│   ├── plugin.py                    # Plugin class (implements CQ Plugin interface)
│   ├── client.py                    # Client for API connections
│   ├── spec.py                      # Plugin configuration spec
│   └── tables/
│       ├── __init__.py
│       └── example_resource.py      # Table definition + resolver
└── tests/
    └── test_example_resource.py     # Unit tests
```

### dp.yaml (CloudQuery Python variant)

```yaml
apiVersion: datapackage.infoblox.com/v1alpha1
kind: DataPackage
metadata:
  name: "<name>"
  namespace: "default"
  version: "0.1.0"
spec:
  type: cloudquery
  cloudquery:
    role: source
    language: python
    tables:
      - "*"
    grpcPort: 7777
```

**Fields**:
| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `spec.type` | string | yes | Must be `cloudquery` |
| `spec.cloudquery.role` | string | yes | Must be `source` |
| `spec.cloudquery.language` | string | yes | `python` or `go` |
| `spec.cloudquery.tables` | string[] | no | Tables to sync (default: `["*"]`) |
| `spec.cloudquery.grpcPort` | int | no | gRPC listen port (default: 7777) |

### Dockerfile (generated, never committed)

A multi-stage Dockerfile generated by the CLI at build time:

```
Stage 1 (builder):
  Base: python:3.11-slim
  Actions: pip install --target=/deps from requirements.txt (with cache mount)
  Output: /deps directory + /app source code

Stage 2 (runtime):
  Base: gcr.io/distroless/python3-debian12:nonroot
  Actions: COPY deps + app, set PYTHONPATH
  Entrypoint: python3 main.py serve --address [::]:7777
```

**Relationship**: One Dockerfile is generated per `dp run` invocation. It is never written to disk in the user's project — it's piped via stdin to `docker buildx build`.

### Template Variables

Variables available to all `.tmpl` files during `dp init` rendering:

| Variable | Type | Source | Example |
|----------|------|--------|---------|
| `{{.Name}}` | string | CLI arg | `foo` |
| `{{.Namespace}}` | string | default | `default` |
| `{{.Role}}` | string | default | `source` |
| `{{.Description}}` | string | derived | `A CloudQuery source plugin` |
| `{{.Version}}` | string | default | `0.1.0` |
| `{{.Language}}` | string | CLI flag | `python` |

## State Transitions

### Plugin Lifecycle (during `dp run`)

```
INIT → BUILD → IMPORT → DEPLOY → READY → [SYNC] → CLEANUP
```

1. **INIT**: CLI detects Python via `pyproject.toml` / `requirements.txt` / `main.py`
2. **BUILD**: Generate Dockerfile, run `docker buildx build`
3. **IMPORT**: `k3d image import` into cluster
4. **DEPLOY**: `kubectl run` as pod with `--image-pull-policy=Never`
5. **READY**: `kubectl wait --for=condition=Ready`, port-forward, gRPC health check
6. **SYNC** (optional): If `--sync`, deploy destination pod, run `cloudquery sync`
7. **CLEANUP**: Delete pod(s), kill port-forward(s)

This is identical to the Go plugin lifecycle — no Python-specific state transitions.

## Validation Rules

- `spec.cloudquery.language` must be `python` or `go`
- `requirements.txt` must exist for Python plugins (checked during build)
- `main.py` must exist for Python plugins (checked during build)
- `pyproject.toml` should specify `requires-python >= "3.12"` (informational, not enforced by CLI)
