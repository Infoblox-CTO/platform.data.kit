# Implementation Plan: Helm-Based Dev Dependencies

**Branch**: `013-helm-dev-deps` | **Date**: 2026-02-15 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/013-helm-dev-deps/spec.md`

## Summary

Replace the ad-hoc, per-service deployment code in the k3d runtime with a uniform Helm chart-based mechanism. Each dev dependency (Redpanda, LocalStack, PostgreSQL, Marquez) is represented as a Helm chart in `sdk/localdev/charts/`, embedded in the CLI binary via `embed.FS`, and deployed through a single `DeployCharts()` function. Upstream charts (Redpanda, PostgreSQL) are used as subcharts; custom charts are maintained for LocalStack and Marquez. Init Jobs seed required resources (topics, buckets, schemas). Chart version and value overrides are supported via the existing config system.

## Phase 2 Summary (for /speckit.tasks)

### Phase 1: Chart Definition Types & Registry
- Define `ChartDefinition`, `PortForward`, `DisplayEndpoint`, `ChartOverride` types in `sdk/localdev/charts/chart.go`
- Define `DefaultCharts` registry in `sdk/localdev/charts/embed.go` with all 4 chart definitions
- Unit tests for type construction and registry validation

### Phase 2: Refactor Redpanda Chart (Upstream Subchart)
- Replace custom Chart.yaml with upstream subchart dependency (redpanda 25.3.2)
- Configure dev-mode values (1 replica, no TLS, no persistence, console enabled)
- Remove custom deployment.yaml and service.yaml templates (delegated to subchart)
- Add init-topics.yaml post-install hook Job
- Run `helm dependency build`, commit Chart.lock and charts/*.tgz

### Phase 3: Refactor PostgreSQL Chart (Upstream Subchart)
- Replace custom Chart.yaml with upstream subchart dependency (bitnami/postgresql 18.3.0)
- Configure dev-mode values (standalone, no persistence, nano preset, auth credentials)
- Add initdb scripts for schema/table creation
- Remove custom deployment.yaml and service.yaml templates
- Run `helm dependency build`, commit Chart.lock and charts/*.tgz

### Phase 4: Enhance LocalStack Chart (Custom, Add Init)
- Bump image tag from 3.0 to 3.8.1 for parity with compose
- Add init-buckets.yaml post-install hook Job for S3 bucket creation
- Keep existing custom deployment.yaml and service.yaml templates

### Phase 5: Create Marquez Chart (Custom, New)
- Create chart directory with Chart.yaml, values.yaml
- Create deployment.yaml (Marquez API server), deployment-web.yaml (Web UI)
- Create service.yaml (API + admin ports), service-web.yaml (web port)
- Create init-job.yaml (create marquez DB in shared PostgreSQL, seed namespaces)
- Add Marquez to embed directive and DefaultCharts registry

### Phase 6: Uniform Deploy Function
- Implement `DeployCharts()` in `sdk/localdev/deploy.go`
- Extract charts from embed.FS to temp dir
- Apply config overrides (version, values)
- Run `helm upgrade --install` in parallel per chart
- Return `DeployResult` with per-chart success/failure
- Unit tests with mock helm execution

### Phase 7: Refactor K3dManager
- Refactor `k3d.go` to use `DeployCharts()` instead of inline chart deployment
- Derive port-forwarding rules from `ChartDefinition.PortForwards`
- Derive health check targets from `ChartDefinition.HealthLabels`
- Remove all hardcoded service lists, port maps, and health targets
- Update `ports.go` to derive `DefaultPorts` from `DefaultCharts`
- Update `portforward.go` to accept `[]PortForward` instead of hardcoded rules
- Update all unit tests

### Phase 8: Config Extension & CLI Updates
- Add `Charts map[string]ChartOverride` to `DevConfig` struct
- Wire config overrides into `DeployCharts()` call path
- Update `cli/cmd/dev.go` to derive endpoint display from `DefaultCharts`
- Remove hardcoded endpoint print block
- Add Marquez ports to port availability check
- Update unit tests and config tests

### Phase 9: Build, Docs & E2E
- Add `helm-deps` Makefile target
- Update docs (getting-started, reference/cli)
- E2E tests for `dp dev up/down/status` with new chart set
- Verify offline operation (SC-004)
- Verify backward compatibility (SC-006)
